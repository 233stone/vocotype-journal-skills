# 日记结构

只有在用户想要一种比默认脚本输出更“成文”的日记风格时，才读这个文件。

## 最小结构

除非用户明确要求换格式，否则保留这些章节：

1. `## 今日概览`
2. `## 今天做了什么`
3. `## 今日状态`
4. `## 时间线`

## 写作规则

- 除非用户明确要反思型文风，否则优先使用短段落或短列表，不要写成长篇抒情文字。
- 活动内容要具体，并且能回到提取出来的数据本身。
- 推断要显式标出来，比如用 `从记录看`、`从数据上看` 这样的表达。
- 如果数据集里没有情绪或能量字段，就明确写 `没有明确状态数据`。
- 重复、低价值的事件可以合并成一句话。
- 不要假装这份数据覆盖了用户完整的一整天。

## 建议的叙事化改写

如果用户想让它更像一篇真正的日记，可以把结构化草稿改写成：

- 一小段开头，概括当天
- 一段正文，讲主要活动
- 一段正文，讲状态、能量或节奏
- 一个很短的关键时间线列表

## 取舍建议

如果这个 skill 被接到定时 automation 里，优先选择稳定的格式，而不是过度追求文风。稳定结构更方便跨天对比，也更方便后续修改提取逻辑时做回归检查。
